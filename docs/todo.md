# AI 线上问题排查 MVP - TODO

## P0 - 必须完成

- [ ] **接入日志平台（至少查错误日志）**
  - 线上排查的第一手数据，没有日志的"初筛卡片"说服力不够
  - 需确认日志平台（ELK/Loki）及 API 接入方式
  - 支持按时间窗口、服务、错误关键词过滤

- [ ] **结构化初筛卡片输出**
  - 当前 AI 输出只有 `possible_causes` 列表，需对齐设计文档定义的 7 个区块：
    1. 异常摘要
    2. 影响服务和关键指标变化
    3. 关键时间线
    4. 可能原因 Top 3
    5. 每个原因对应的证据
    6. 建议下一步排查动作
    7. 信息缺口
  - AI prompt 和返回解析需严格对齐卡片结构

## P1 - 重要

- [ ] **接入监控指标（错误率/RT/QPS）**
  - 影响面判断的基础数据
  - 需确认监控平台（Prometheus/Grafana）及查询方式
  - 支持按时间窗口、服务过滤

- [ ] **用户输入参数传入��析链路**
  - `environment`、`time_window`、`problem_type`、`services` 字段已存在于 API 请求模型中，但未被消费
  - 需将这些参数传入 `jira_analyzer.py` 的 `analyze()` 方法，用于：
    - 按时间窗口裁剪日志/监控数据
    - 按环境过滤数据源
    - 按服务列表圈定查询范围

- [ ] **上下文预处理/摘要层**
  - 当前实现将原始数据直接拼进 prompt，真实场景数据量大会超 token 限制
  - 需在调 AI 之前加一层数据预处理：
    - 日志聚类/摘要
    - 异常指标提取
    - Trace 关键路径摘要
    - 发布变更摘要

## P2 - 可延后

- [ ] **接入发布记录**
  - 变更导致的问题占比很高
  - 需确认发布系统及 API 接入方式
  - MVP 阶段可先支持手动填写

- [ ] **反馈接口 + 效果度量**
  - MVP 验证需要收集：是否有帮助、是否命中根因
  - 需新增反馈 API 接口和存储
  - 记录请求开始/结束时间用于度量定位耗时
  - 可先用简单方案（数据库/文件记录）
