# MVP 优化讨论清单

本文档用于记录本地 MVP 后续逐步讨论和确认的优化项。每个优化项先描述目标、范围、暂不做内容和待确认问题，确认后再进入实现。

## 优化项 1：从 Jira 文本中自动识别 Trace ID

### 背景

当前 MVP 希望尽量做到用户只输入 Jira 链接即可启动分析。Trace ID 是拉取运行时链路和 SQL 证据的关键输入，但如果要求用户额外手动填写，会打断“只输入 Jira”的最短链路。

为了本地演示，可以设计受控 Jira 案例：在 Jira 描述中明确写入 Trace ID。系统读取 Jira 后自动提取 Trace ID，再继续拉取 Trace 数据。

### 目标

当用户没有手动传入 Trace ID 时，系统尝试从 Jira 文本信息中自动识别 Trace ID。

识别成功后，系统自动使用该 Trace ID 拉取 Trace 运行时证据；识别失败时，再提示用户补充 Trace ID。

### 建议识别来源

第一版建议只从文本字段中识别：

- Jira 标题
- Jira 描述
- 线上问题描述字段
- Jira 评论

### 建议支持格式

第一版优先支持明确标记的写法，例如：

```text
Trace ID: 1774764304798_AKFw
traceId=1774764304798_AKFw
traceid：1774764304798_AKFw
链路ID：1774764304798_AKFw
```

Trace ID 本身可以先按当前样例格式识别：

```text
13 位时间戳 + _ + 字母数字
```

示例：

```text
1774764304798_AKFw
```

### 使用优先级

建议规则：

1. 如果用户/API 请求中已经传入 Trace ID，优先使用用户传入值。
2. 如果用户没有传入 Trace ID，则从 Jira 文本中识别。
3. 如果识别到一个 Trace ID，自动使用。
4. 如果识别到多个 Trace ID，第一版可以先返回候选让用户选择，或临时使用第一个并在结果中说明。
5. 如果没有识别到 Trace ID，提示用户补充。

### 暂不做

第一版暂不处理：

- 图片 OCR
- 附件图片识别
- 截图中的 Trace ID 提取
- 复杂语义判断
- 多个 Trace ID 的智能排序

### MVP 价值

这个优化可以把主链路从：

```text
用户输入 Jira + 用户填写 Trace ID
```

收敛为：

```text
用户输入 Jira → 系统自动识别 Trace ID
```

它能显著降低演示时的操作成本，并强化 MVP 的闭环感。

### 待确认问题

- Jira 评论是否必须纳入第一版识别范围？
- 多个 Trace ID 时，是返回候选给用户选，还是默认使用第一个？
- Trace ID 是否只支持当前样例格式，还是需要兼容更多格式？
- 识别到 Trace ID 后，是否需要在页面上展示“系统已识别到 Trace ID”？

## 优化项 2：取消 API path 作为代码分析前置条件

### 背景

之前流程中曾把“是否有 API path / 接口名”作为拉取代码并分析调用链的前置判断。但在本地 MVP 中，这不应作为必要条件。

当前演示链路可以先通过 Jira 获取问题描述，再通过 Trace ID 拉取链路日志。链路日志中通常可以包含 HTTP 服务节点、接口路径、错误节点、SQL 节点和相关服务信息。系统可以从 Trace 运行时证据中提取 API path 或代码搜索关键词，再去分析固定服务代码。

### 目标

用户不需要手动输入 API path / 接口名，系统也能继续做代码分析。

系统优先从 Trace 链路日志中提取接口路径、错误节点、服务名、SQL 表名和业务关键词，用这些线索去搜索和分析固定服务代码。

### 建议流程

```text
1. 用户输入 Jira 链接
2. 系统读取 Jira，收集问题描述和 Trace ID
3. 系统拉取 Trace 链路日志
4. 系统从 Trace 中提取 API path、错误节点、服务名、SQL 和关键词
5. 系统读取或拉取固定服务代码
6. 系统基于 Trace 线索搜索相关代码
7. 系统综合 Jira、Trace 和代码证据生成初筛卡片
```

### 代码分析线索优先级

建议优先级：

1. Trace 中提取到的 API path / HTTP 节点
2. Trace 中的错误节点、异常文本、服务名
3. Trace 中的 SQL 表名和 SQL 关键词
4. Jira 描述中的异常关键词、类名、业务词
5. 固定服务代码的默认搜索范围

### 暂不做

第一版暂不要求用户填写：

- API path
- 接口名
- Controller 名称
- 方法名

如果用户填写了，也可以作为增强线索；但不填写不应阻断主流程。

### MVP 价值

这个优化可以把用户输入进一步收敛到：

```text
Jira 链接
```

系统通过 Jira 找 Trace ID，再通过 Trace 反向获得代码分析线索，减少用户对接口、服务和代码结构的理解成本。

### 待确认问题

- 如果 Trace 中没有提取到 API path，是否仍然继续用错误节点和 SQL 表名搜索代码？
- 固定服务代码是否始终作为第一版默认分析对象？
- 页面是否隐藏 API path 输入框，还是保留为可选高级字段？

## 优化项 3：代码分析统一走本地 Claude Code CLI

### 背景

当前项目已有一套基于项目结构和命名约定的 Java 静态调用链分析逻辑，用于尝试定位 Controller、Service、DAO、Mapper 和 MyBatis SQL。但这条路径依赖较多固定约定，也容易在真实代码中遇到动态调用、接口实现、多态、AOP、反射、跨模块等复杂情况。

本地 MVP 的核心目标不是验证自研静态分析器是否足够完整，而是验证 Jira、Trace、SQL 和代码证据能否帮助研发更快形成初筛判断。因此第一版代码问题分析可以统一交给本地 Claude Code CLI 执行。

### 目标

代码问题分析必须调用本地 Claude Code CLI。

系统将 Jira 信息、Trace 证据、SQL 证据和固定服务仓库路径整理成分析上下文，交给 Claude Code CLI 在本地代码仓库中做只读分析，并返回结构化结论。

### 建议流程

```text
1. 用户输入 Jira 链接
2. 系统读取 Jira 信息
3. 系统识别或获取 Trace ID
4. 系统拉取 Trace 和 SQL 运行时证据
5. 系统准备代码分析上下文
6. 系统调用本地 Claude Code CLI
7. Claude Code CLI 在固定服务代码仓库中只读分析
8. 系统接收结构化代码分析结果
9. 系统生成初筛卡片
```

### 输出要求

Claude Code CLI 不应只返回自由文本，建议返回结构化结果，例如：

```json
{
  "summary": "问题判断摘要",
  "possible_causes": [
    {
      "cause": "可能原因",
      "confidence": "high | medium | low",
      "evidence": [
        {
          "file": "相关文件路径",
          "method": "相关方法名",
          "reason": "为什么该代码与问题相关"
        }
      ]
    }
  ],
  "related_files": ["相关文件路径"],
  "next_steps": ["建议人工继续确认的点"],
  "missing_info": ["还缺少的信息"]
}
```

### 执行约束

第一版建议固定以下约束：

- 固定工作目录为预配置服务仓库，例如 `./repos/super-mario`。
- 只做只读分析，不自动改代码。
- 优先允许读取和搜索代码，不允许写文件。
- Prompt 中必须带上 Jira 摘要、Trace 错误节点、SQL 证据和必要关键词，避免全仓无目标扫描。
- 分析结论必须包含证据路径、方法名、理由和置信度。

### 不支持降级

第一版不支持回退到旧的自研静态调用链分析。

如果 Claude Code CLI 不可用、未登录、超时、权限不足或执行失败，系统应明确返回代码分析失败原因，并把它作为初筛卡片的信息缺口，而不是切换到旧分析路径。

### 暂不做

第一版暂不处理：

- 自动修改代码
- 自动提交代码
- 自动跨仓库选择
- 交互式 Claude Code 会话
- 旧静态调用链分析兜底
- 多种分析引擎自动切换

### MVP 价值

这个优化可以显著降低自研静态分析成本，把代码分析能力聚焦到“读代码、找证据、给判断”。

主链路从：

```text
API path → Controller → Service → DAO → Mapper → SQL
```

调整为：

```text
Jira + Trace + SQL 证据 → Claude Code CLI 只读分析固定代码仓库 → 结构化代码结论
```

这更符合本地 MVP 快速验证排障价值的目标。

### 待确认问题

- Claude Code CLI 的工作目录是否固定为 `./repos/super-mario`？
- CLI 输出是否必须是 JSON，还是允许 Markdown 后端再包装？
- CLI 超时时间设置为多少合适？
- 是否需要在页面上展示“代码分析由本地 Claude Code CLI 完成”？

## 优化项 4：固定代码分析版本，不自动拉取最新代码

### 背景

本地 MVP 演示通常会选择一个历史线上问题案例。这个案例对应的 bug 很可能已经在最新代码中被修复。

如果系统在分析时自动拉取最新代码，Claude Code CLI 看到的就可能是修复后的代码，而不是事故发生时的代码。这会导致 Jira、Trace 和代码版本不一致，影响分析结论。

因此演示型 MVP 需要固定代码分析版本，让代码仓库停留在 bug 修复前的提交。

### 目标

确保 Claude Code CLI 分析的是与 Jira/Trace 对应的历史事故现场代码，而不是最新代码。

第一版使用人工准备好的本地仓库目录，由人工提前 checkout 到 bug 修复前的 commit。系统分析过程中只读取该目录，不执行自动更新。

### 建议流程

```text
1. 人工选择演示 Jira 案例
2. 人工确认该问题对应的 bug 修复前 commit
3. 人工 clone 固定服务仓库到本地
4. 人工 checkout 到 bug 修复前 commit
5. 系统配置该本地仓库目录为 Claude Code CLI 工作目录
6. 用户输入 Jira 链接启动分析
7. 系统读取 Jira、拉取 Trace/SQL 证据
8. Claude Code CLI 只读分析固定版本代码
9. 系统输出线上问题初筛结论
```

### 版本规则

第一版建议固定以下规则：

- 不自动 `git pull`
- 不自动 `git fetch`
- 不自动 checkout 最新分支
- 不根据 Jira 自动判断代码版本
- 不在页面上要求用户选择 commit
- 由人工提前准备本地仓库和指定 commit
- 系统只读取该固定目录

### 与现有仓库选择的关系

当前系统支持 `repo_url + ref` 方式指定分支或 commit，也支持通过 `repo_key` 查找仓库配置。

但为了演示稳定，第一版不依赖实时拉取和自动切换版本，而是优先使用已经准备好的本地仓库目录。

### MVP 价值

这个优化保证三类信息处于同一个事故现场：

```text
Jira 描述的问题
Trace 记录的请求
bug 修复前的代码版本
```

它可以避免因为代码已经修复而导致 Claude Code CLI 无法复现或解释当时的问题。

### 暂不做

第一版暂不处理：

- 自动根据 Jira 时间推断 commit
- 自动根据修复 PR 查找父提交
- 自动切换分支或 commit
- 自动更新本地仓库
- 多版本代码对比分析

### 待确认问题

- 固定本地仓库目录是否仍使用 `./repos/super-mario`？
- 是否需要在页面或结果中展示当前代码 commit？
- Claude Code CLI 分析前是否需要校验当前目录确实处于预期 commit？
